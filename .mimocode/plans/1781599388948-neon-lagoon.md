# 分析：为什么一个任务失败后，其他任务不会继续执行

## 问题

用户提供的 YAML 配置中，`on_failure: fail` 导致一旦某个任务失败，流水线立即停止，后续任务全部被跳过。

## 根本原因

### 1. 配置本身指定了 `on_failure: fail`

在用户提供的 YAML 中，顶级 config 设置了 `on_failure: fail`。这是**默认值**，本质上是显式声明了"失败即停止"的行为。

### 2. 执行引擎的逻辑路径

关键代码位于 `server/taskpps/engine/runner.py`，有两个层面的检查：

#### 子流水线层面（第 287-288 行）
```python
on_failure = sub.config.on_failure
if on_failure != "continue":
    # 将下游子流水线标记为失败（SKIPPED）
```
用户的 pipeline 只有一个隐式子流水线（所有任务在同一层级），所以子流水线层面不会阻断。

#### 任务层面（第 446-452 行）
```python
on_failure = task.on_failure or sub.config.on_failure or "fail"
# ...
should_skip = on_failure != "continue"
```
这是**关键决策点**。当 `on_failure` 为 `"fail"` 时：
- 如果一个任务失败，所有**直接或间接依赖于**它的任务都会被标记为 `SKIPPED`
- 由于 `execution_strategy: sequential`，任务顺序执行，失败的任务之后的所有任务都不会执行

### 3. YAML 中已注释掉的任务

配置中被 `#` 注释的任务（AOSP、APP、APP_sc、BT、Power Management、Security）根本不会出现在解析后的 pipeline 结构中，不会被加载或执行。

## 解决方案

### 选项 A：将 `on_failure` 改为 `continue`

```yaml
config:
    on_failure: continue
```

这样即使某个任务失败，后续任务仍然会继续执行。失败的任务会在流水线记录中标记为 `FAILED`，但不会阻断其他任务。

### 选项 B：为特定任务设置 `on_failure: continue`

```yaml
tasks:
    - name: Debugtrace
      on_failure: continue
      # ...
```

只有该任务失败时不会阻断流水线，其他任务仍然遵循顶级的 `fail` 策略。

修改后可以通过以下方式验证：
1. 启动 test 中的场景测试：`server/tests/scenario/test_pipeline_execution.py` 中的 `test_on_failure_continue`
2. 或运行实际的 pipeline 观察行为
